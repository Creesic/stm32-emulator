## Key Learnings

- **Proteus F7 boot and idle:** A 4 KiB system-ID page at 0x1FF0F000 is required for the image to pass its UID reads. Normal Unicorn returns expose even PCs; always re-enter Cortex-M through the Thumb-bit address. The image then reaches its WFI idle loop without an instruction error.
- **Proteus F7 transport evidence:** The observed image does not access either OTG-FS or OTG-HS during full boot. Its matching board.mk enables USART2/K-line serial but no USB/OTG definitions, so a USB CDC TCP endpoint would not be firmware-backed for this image.
- **Correction, 2026-07-10:** The user confirms the current Proteus F7 binary is USB-enabled. Treat the absence of live OTG traffic before a virtual host attach as insufficient to infer USB is disabled; model the virtual USB host/TCP attach path first.
- **Proteus F7 USB power:** The F7 board header sets `BOARD_OTG_NOVBUSSENS TRUE`; physical VBUS is intentionally ignored because this board is device-only. MCU reset and the virtual host attach are sufficient—do not add a VBUS GPIO gate.
- **Core peripherals outside the SVD:** The F767 SVD omits SysTick and SCB. Register SysTick at 0xE000E010 and SCB at 0xE000ED00 explicitly; otherwise scheduler interrupts and PendSV requests are silently dropped.
- **Static EpicEFI catalog:** Generate from every firmware/config/boards board.yaml at development time. Preserve unique source IDs; qualify duplicate IDs with their relative board path. The current source produces 61 catalog entries.

- **Project:** stm32-emulator

- **Unicorn 2.1.5 migration:** The binding uses unicorn_const::Prot rather than Permission, and mem_map/mmio_map lengths are u64. The only production mapping call sites are in src/system.rs.
- **Unicorn 2.1.5 lifetime:** A helper returning a newly created no-data engine must name the owned type as Unicorn<'static, ()>; local variable type elision still works elsewhere.
- **Startup model:** Proteus F767 initial clock bring-up requires stateful RCC CFGR/CSR readiness and PWR CR1/CSR1 voltage and over-drive readiness. The next observed blocker is the unmodeled FLASH ACR latency register.
- **Cortex-M7 engine mode:** Unicorn 2.1.5 treats deprecated `Mode::MCLASS` as a forced Cortex-M33 selection during ARM CPU initialization, overriding `ctl_set_cpu_model`. Create the engine in Thumb mode and select Cortex-M4 or Cortex-M7 only through the control API; the CPU model itself supplies M-profile semantics.
- **Current Proteus execution boundary:** With the explicit Cortex-M7 model no longer overridden, the unmodified firmware executes both VDIV.F32 at 0x002397ec and the following double-precision conversion at 0x002397f0, reaching 0x002397f4. USB peripheral work may resume from this verified boundary.
- **Proteus startup readiness:** Before any later runtime path, this image requires RCC BDCR LSE-ready (bit 1 after LSEON) and PWR CSR1 bit 3 after the observed CSR1 bit-9 enable write. Both are now trace-backed state models; USB still has no live OTG access through 5M instructions.

- **Virtual USB root cause (2026-07-10):** The TCP bridge itself works (live-tested: connecting to 127.0.0.1:29000 logs "USB CDC TCP client connected" and "Virtual USB host attached"). Nothing happens after that because Proteus F7 firmware never touches any OTG_FS register at all (0 accesses across 100M+ traced instructions) — execution is permanently parked at pc=0x002793e0 (RTOS idle loop) driven only by SysTick; PendSV (irq=-2) never fires once, so the emulated ChibiOS scheduler never switches away from the idle thread. epicefi_fw source confirms `EFI_USB_SERIAL=TRUE` by default for F7 and `runEpicEfi()` unconditionally calls `startUsbConsole()` near boot (epicefi.cpp:220-221) — but that only creates a `UsbThread` (PRIO_CONSOLE); the actual OTG_FS-touching code (`usb_serial_start()`/`usbStart()`) only runs once that thread is scheduled, which never happens here. This is a scheduler/context-switch gap, not a firmware-USB-disabled issue and not (yet) an OtgFs peripheral bug.
- **OtgFs model is a stub beyond the reset bit:** `peripherals/otg_fs.rs` only implements GRSTCTL/GINTSTS/GINTMSK. `modeled_range` expands its claimed window to 8KB, so OTG_FS_HOST/DEVICE/PWRCLK accesses would fall into a semantics-free BTreeMap (no DCFG/DCTL/DIEPCTLx/endpoint/FIFO behavior). `UsbCdcTcp::push_from_device`/`take_for_device` are dead code — never called from `OtgFs` — confirmed by a `cargo build` dead-code warning. Even if the scheduler gap above were fixed, real CDC enumeration/data transfer would still need the fuller DWC2 model described in `docs/superpowers/specs/2026-07-10-proteus-f7-virtual-usb-design.md`.
- **Scheduler stall pinpointed further (2026-07-11), see [[bug-075]]:** Used Ghidra (raw ARM:LE:32:Cortex import of rusefi.bin @ 0x00200000, no symbols) plus temporary Unicorn memory-read instrumentation in emulator.rs (added and fully reverted) to confirm ChibiOS's tick/timer machinery is NOT broken: `systime` at RAM 0x200231b8+0x1c increments correctly every SysTick; the virtual-timer list at +0x10 is processed correctly; a real ~12-13 tick boot timer (object 0x20021578, callback 0x0027987c = `_thread_timeout`-equivalent) fires on schedule (clk~3,024,913) and correctly re-inserts its thread into what's clearly a priority-sorted ready list via 0x0020b170 (walks-and-inserts pattern, i.e. real `chSchReadyI`). But no PendSV/context-switch ever follows. The ISR epilogue chain (0x0020d830→0x28214c→0x282690→0x25fc74) is gated by a byte flag at RAM 0x2005794d that reads 0 on every sample across the whole boot; **patched it out live (NOP at 0x25fc7a via config `patches:`) and behavior did not change** — still falls through to a dead end at 0x25fccc, 0 PendSV events. This disproves "that flag gates the reschedule trigger" — it's more likely an inert debug/profiling hook (maybe SEGGER SystemView, which the repo vendors bindings for and no-ops when not attached). The actual `chSchRescheduleS`/priority-compare-and-switch call site is still unlocated. Next real step needs a symbol-matched firmware build (blocked on [[bug-062]], missing ARM toolchain) rather than more raw-disassembly guessing.
- **Scheduler stall root cause FOUND AND FIXED (2026-07-11), see [[bug-076]]:** The original firmware source is buildable from `C:\Users\Tera\Documents\GitHub\epicefi_fw` — `./build.sh build proteus proteus_f7 firmware` (from `firmware/`) via a Gradle→make pipeline. Needed: WSL Ubuntu-22.04, `zip`/`unzip`, and **GCC ≥11.3.1** (the version check is hard-enforced) — Ubuntu 22.04's `gcc-arm-none-eabi` apt package is only 10.3.1, so download `arm-gnu-toolchain-12.3.rel1-x86_64-arm-none-eabi` directly from developer.arm.com and prepend its `bin/` to PATH (must be double-quoted, `PATH="$HOME/opt/.../bin:$PATH"` — an unquoted value breaks on the parens in "Program Files (x86)" inherited from Windows). Also: `rm -rf firmware/build firmware/.dep` before switching toolchain versions, or you get an LTO version-mismatch link error from stale objects. The resulting `build/epicefi.elf` (with `-ggdb -g`) has its vector table SP/reset match the known rusefi.bin exactly, confirming it's a faithful rebuild despite a ~12KB size difference (probably just shifted addresses from unrelated changes). Using `arm-none-eabi-addr2line`/`nm` against this ELF: the actual bug is in **our own emulator**, `src/peripherals/scb.rs` — `Scb::read()` never implemented ICSR (offset 0x0004), always returning 0. ChibiOS's ARMv7-M port (`__port_irq_epilogue`, `ChibiOS/os/common/ports/ARMv7-M/chcore.c:312`) only performs its post-IRQ thread switch when `SCB->ICSR` bit 11 (RETTOBASE) reads set; with our SCB always reporting 0, that check always failed and the switch was always skipped — for every thread, not just USB. Fixed by returning `ICSR_RETTOBASE` unconditionally (correct since this NVIC never nests interrupts, so every return already is to base level). Confirmed via a live re-run of the original rusefi.bin: the firmware now proceeds past the point it always stalled at before.
- **SVCall blocker also FIXED (2026-07-11), see [[bug-077]]:** This ChibiOS ARMv7-M port performs its actual ISR-context switch via a real `svc 0x0` instruction (disassembly at `0x00200ca2`, matches `__port_switch_from_isr` in the symbol-matched build) — not PendSV. Fixed by adding `irq::SVCALL = -5` and `Nvic::enter_svcall()` (reuses the existing `run_interrupt` push-context/vector-jump path, since `IRQ_OFFSET + (-5) = 11` lands on the SVCall vector-table word), wired from `emulator.rs`'s intr_hook on Unicorn exception 2 (`EXCP_SWI`). Live-verified on the unmodified rusefi.bin: no more crash, and the resuming PC varies across ticks for a while afterward — real evidence of multiple threads actually executing, not just one idle loop.
- **bug-078 corrected by user, then root-caused (2026-07-11):** My first theory ("threads legitimately blocked on unsimulated crank/CAN/ADC input") was wrong and the user correctly overruled it — EpicEFI/rusEFI brings up its USB console standalone on a bench, no car connection needed. Re-investigated with a temporary chSysHalt diagnostic: the resting address (0x00234a90 in one build) is `__NVIC_SystemReset`'s post-AIRCR spin loop — the firmware actually faulted and tried to reset the MCU, which our Scb doesn't implement, so it spins forever instead of rebooting. Captured the real halt reason live: `chSysHalt("SV#8")` from `__dbg_check_enter_isr` (chdebug.c:220) — a later ISR prologue finds the OS lock still held from an earlier critical section.
- **bug-078 actually fixed + two more blockers found and fixed, see [[bug-082]] (2026-07-11):** Fixed the 104-vs-100-byte extended-frame gap (real fix, kept) but proved via an identical clk-for-clk repro that it was NOT the SV#8 cause. The real cause: `chSysLock()`/`chSysUnlock()` and `__port_irq_epilogue` mask interrupts via **BASEPRI** (disassembly: `movs r3,#48; msr BASEPRI,r3` ... `movs r3,#0; msr BASEPRI,r3`), not PRIMASK — but `Nvic::are_interrupts_disabled` only ever checked PRIMASK, so SysTick could fire mid-critical-section anywhere in the firmware, corrupting the lock bookkeeping. Fixed by also checking BASEPRI (nonzero = masked; we don't model priorities, so this is an all-or-nothing approximation, and `enter_svcall` is unaffected since it's called directly, not through this gate — matching how this port configures SVCall above any BASEPRI it raises). This alone got execution past the reset-spin permanently. Two more blockers surfaced immediately after, each pinned via the symbol-matched build and fixed the same way: (a) `chSysPolledDelayX` polls `DWT->CYCCNT` (confirmed via the literal pool value `0xe0001000`, DWT base) which we never modeled at all (not in the SVD, like SysTick/SCB) — added a minimal `Dwt` peripheral (`src/peripherals/dwt.rs`) whose CYCCNT tracks `NUM_INSTRUCTIONS`; (b) `otg_core_reset` polls `GRSTCTL_CSRST`/`TXFFLSH`/`RXFFLSH` (self-clearing W1-to-request bits) which `OtgFs::register_write` stored verbatim instead of completing instantly — masked them out on write. **Live-verified end to end**: unmodified rusefi.bin + a live TCP host attach now runs a real USB reset ISR — 3.6M+ real OTG_FS register accesses (GRXFSIZ, endpoint 0 CTL, GINTMSK set to a real nonzero mask, DOEPMSK/TSIZ, FIFO sizing), then correctly waits for the next real USB protocol event. That next-event gap is exactly the pre-existing "OtgFs is a stub beyond the reset bit" finding — full DWC2 enumeration/FIFO modeling is the remaining, separately-scoped work.
- **Gotcha:** `register_core_peripherals()` (SysTick/SCB/DWT, none SVD-declared) pushes directly into the sorted `peripherals` Vec without sorting itself — `get_peripheral` binary-searches it. New entries MUST be pushed in ascending start-address order or lookups for entries after the out-of-order one silently break (caught by `core_scb_model_covers_interrupt_control_register` failing when DWT was appended after SCB despite having a lower address).

## Decision Log

- **Proteus F7 FPU:** Upgrade to upstream unicorn-engine 2.1.5 and select CORTEX_M7 through its CPU-model control. Retain CPACR for architectural fidelity; do not hook or emulate floating-point instructions in Rust.
- **Proteus F7 VFP continuation:** Retain Unicorn 2.1.5; remove deprecated MCLASS mode so its CPU-model control instantiates the intended double-precision Cortex-M7 rather than forcing Cortex-M33.
- **Proteus F7 USB:** Keep the supplied binary. Implement the CDC-over-TCP virtual host path instead of replacing or rebuilding the firmware.
